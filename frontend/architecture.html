2 RAG Development: Solving Hallucinations

Building a demo RAG is easy. Building one that actually answers complex questions accurately is incredibly hard. Here is the evolution of my backend logic.

The Problem: Naive RAG is Blind

Initially, I built this using the standard approach: take the user's query, embed it, run a cosine similarity search on ChromaDB, dump the top 5 chunks into an LLM, and ask for an answer.

It failed miserably in two specific scenarios:
1. When users asked for specific acronyms or document IDs (e.g., "Form 1040-X"). Vector math captures "meaning", not exact spelling, so it retrieved the wrong forms.
2. When users asked multi-part questions. The LLM would just hallucinate the parts the vector search missed.

Solution 1: Hybrid Retrieval (Chroma + BM25)

To fix the keyword matching problem, I realized I couldn't rely solely on embeddings. I implemented BM25, a sparse keyword search algorithm.

Now, when a query comes in, my Python backend runs a vector search (Chroma) to find semantic meaning, AND a BM25 search to find exact text matches. It mathematically fuses these results using Reciprocal Rank Fusion (RRF). This completely solved the issue of missing specific part numbers and acronyms.

Solution 2: Multi-Agent Orchestration (LangGraph)

To fix the multi-part question problem, I threw away my linear LangChain code and rewrote the core using LangGraph. I created a "state machine" of distinct agents:

  • Query Planner: Rewrites sloppy user questions into structured search queries.
  • The Critic Agent: This was my biggest breakthrough. After retrieving documents, an independent LLM prompt acts as a "judge". It reads the docs and asks, "Is this enough to answer the user?" If not, it loops back and forces the system to search again with a different query.

By using FastAPI and Groq (which generates tokens blazingly fast), these multiple loops happen in seconds, ensuring the final output is 100% grounded in facts.

3 Infrastructure: Surviving Production

Developing locally with uvicorn main:app --reload using WatchFiles is a great developer experience. However, putting that directly on the internet is a disaster waiting to happen. Here is my production deployment philosophy.

Why Docker? Fixing "It works on my machine"

When I first tried moving my Python code to a server, I ran into endless dependency conflicts. Different Python versions, missing C++ build tools for ChromaDB, etc.

I wrote a comprehensive Dockerfile that packages the OS, Python, Uvicorn, FastAPI, and all ML dependencies into a single immutable image. Now, deployment is just docker run. I use Docker volumes to ensure my Chroma vector database persists even if the container reboots.

Why AWS EC2 over Vercel/Heroku?

Managed services are easy, but they abstract away the system layer and get incredibly expensive when dealing with persistent file storage (like local vector DBs) and long-running websocket/async tasks.

By provisioning my own AWS EC2 instance, I have total control over the Linux environment. It proves I can manage memory, monitor CPU usage, and build cost-effective infrastructure from scratch.

Nginx & fail2ban: The Hostile Internet

You don't realize how hostile the internet is until you tail your EC2 auth logs. Within hours of booting my server, I saw automated scripts from random IP addresses trying to brute-force my SSH port (Port 22).

Here is how I secured the system:
1. Nginx Reverse Proxy: Uvicorn is not designed to face the public internet securely. I put Nginx in front of it to handle SSL/TLS encryption, serve static files, and rate-limit aggressive requests.
2. fail2ban: I installed this service to actively monitor my Linux logs. If an IP fails to log in 5 times within 10 minutes, fail2ban automatically updates the firewall rules to drop all traffic from that IP.
